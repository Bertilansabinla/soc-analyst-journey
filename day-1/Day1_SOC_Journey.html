PART 1

Before I Could Even Start — Everything Was Full

First task: update Kali Linux. Standard practice before starting any new work. Ran sudo apt upgrade and immediately hit:

Warning: More space needed than available: 1,152 MB > 330 MB, installation may fail
Error: You don't have enough free space in /var/cache/apt/archives/

My Kali VM had 29GB of total disk space and was sitting at 98% full with only 653MB available. Before I'd even started.

I tried to clean up. sudo apt clean helped a little. sudo apt autoremove --purge found nothing. Tried clearing journal logs:

sudo journalctl --vacuum-size=100m   # WRONG — lowercase m fails silently
sudo journalctl --vacuum-size=100M   # CORRECT — uppercase M required
🔑 Small detail, real cost: On Linux, case matters in command flags. Lowercase m and uppercase M mean different things. When you're new, small details like this cost you time.

Running sudo fdisk -l /dev/nvme0n1 revealed why disk expansion was blocked:

Device            Start      End  Sectors  Size Type
/dev/nvme0n1p3  2035712 63633407 61597696 29.4G Linux filesystem  ← Kali lives here
/dev/nvme0n1p4 63633408 67106815  3473408  1.7G Linux swap         ← This blocks expansion

Even though VMware had allocated more space, the swap partition (p4) was physically sitting between Kali's partition (p3) and the new free space. You can't grow p3 sideways when there's another partition in the way.

⚠️ Kali disk expansion: unresolved. Would require deleting the swap partition, expanding p3, recreating swap — a more involved operation noted for later.

Mac fix: Ran tmutil deletelocalsnapshots / to clear invisible APFS Time Machine snapshots macOS creates automatically. Freed ~8GB.

PART 2

What Is a SIEM and Why Does It Matter?

SIEM = Security Information and Event Management. The central nervous system of a SOC. Every device generates log files all day — logins, file changes, network connections, process creations. Millions of lines daily across hundreds of machines. No human can read raw logs from hundreds of machines simultaneously. So companies send all logs to one central searchable system — the SIEM. Analysts query that data to find patterns indicating something bad is happening.

Splunk is one of the most widely used commercial SIEMs. Learning it directly maps to what job descriptions ask for.

My plan:

1. Set up Ubuntu Server as the SIEM server → 2. Install Splunk → 3. Load sample security log data → 4. Practice SPL queries → 5. Connect Kali and Windows VMs to send real logs.

Simple on paper.

PART 3

The Internet Problem

Needed to download Splunk onto the Ubuntu VM. The VM was on a host-only network (isolated from the internet — good security practice for a lab). Gave it temporary internet access and confirmed connectivity:

ip a          # IP was there — ens160 with 192.168.114.128
ping google.com   # ping: google.com: Temporary failure in name resolution

The machine had a network connection but no DNS. DNS translates domain names (google.com) into IP addresses. Without it, you can connect to the network but can't find anything by name.

Fix:

echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf

Manually added Google's DNS server (8.8.8.8) to the system's DNS configuration. Internet worked.

PART 4

The Architecture Problem — The Wall I Hit Repeatedly

The ARM vs x86 problem:
MacBook Air M1 = ARM processor. VMs on M1 Mac = also ARM (they have to be — the hardware is ARM). Splunk Enterprise for Linux = compiled for x86 (Intel/AMD) only. There is no ARM64 Linux version of Splunk. You simply cannot install the standard Linux Splunk package natively on an M1 Mac's VMs.

I didn't know this going in. I found out through a series of failures.

Docker as the workaround: Docker lets you run containers. With QEMU emulation, you can theoretically run x86 containers on ARM hardware. Installed Docker, tried:

sudo docker run -d --platform=linux/amd64 -p 8000:8000   -e SPLUNK_START_ARGS='--accept-license'   -e SPLUNK_PASSWORD='splunk123!'   --name splunk-enterprise splunk/splunk:latest

Image downloaded — 1.35GB. Then failed with "no space left on device." The Docker image had filled Ubuntu's remaining space. Ubuntu was 100% full.

PART 5

Expanding Ubuntu's Disk — The First Real Win ✅

Unlike Kali, Ubuntu used LVM (Logical Volume Manager). LVM is more flexible than standard partitioning — you can expand it while the system is running.

sudo vgdisplay
# Output: Free PE / Size: 1873 / 7.32 GiB — unallocated space available to use

Had to type these commands manually — copy-paste between Mac and VM not working yet (more on that shortly):

sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
sudo resize2fs /dev/ubuntu-vg/ubuntu-lv
✅ Result: Ubuntu went from 9.8GB at 100% full → 17GB with 8.8GB free. First real win of the day.
🔑 Lesson: LVM is better than standard partitioning for lab environments. Ubuntu's LVM let me expand in two commands while running. Kali's standard partition layout required a more complex procedure because a swap partition was in the way. When setting up VMs for labs, use LVM.
PART 6

Docker Tries Again — And Keeps Failing

With space available, tried Splunk in Docker again. Downloaded fully but container kept exiting immediately.

Error 1: Splunk updated license terms. Old --accept-license flag wasn't enough. Required new SPLUNK_GENERAL_TERMS environment variable. Updated, deleted old container, tried again.

Error 2 — the real ARM emulation failure:

sudo: effective uid is not 0, is /usr/bin/sudo on a file system
with the 'nosuid' option set or an NFS file system without root privileges?

Splunk's Docker container uses Ansible internally to set itself up. Ansible uses sudo. When running x86 containers on ARM via QEMU emulation, the emulation layer doesn't properly handle kernel-level privilege operations that sudo relies on. Splunk's internal setup fails before it even starts.

Tried --privileged flag → still failed. Tried --user root → still failed. Tried installing proper QEMU binfmt support → still failed.

Conclusion: Running Splunk Enterprise via Docker on ARM Ubuntu VM doesn't work due to how QEMU handles sudo privilege escalation in Splunk's Ansible provisioner. Dead end.
PART 7

The Copy-Paste Problem

Running parallel to all of this: couldn't copy and paste between my Mac and the Ubuntu VM terminal. The normal Linux terminal paste shortcut Ctrl+Shift+V didn't work. Right-clicking showed a greyed-out Paste option. VMware's Edit menu operated at the VMware level, not inside the terminal.

Solution: SSH

Instead of the VM's graphical terminal, connected from Mac Terminal directly to Ubuntu:

ssh administrator@172.16.23.135
✅ Inside native Mac Terminal = full Mac copy-paste with Cmd+V. Problem solved permanently.

Setup required on Ubuntu first:

sudo apt install openssh-server -y
sudo systemctl enable ssh
sudo systemctl start ssh
🔑 Lesson: SSH into your VMs from your host terminal immediately. Don't fight the VM's graphical terminal for copy-paste. It's faster, cleaner, and more realistic to how you'd actually administer a server in production.
PART 8

The Right Solution Was There All Along

After hours of Docker attempts, I finally searched differently and found the answer I should have found first:

✅ Splunk Enterprise has a native macOS ARM build.

Go to splunk.com/download → click Mac OS tab (not Linux) → ARM — macOS — .dmg — 840.8 MB

Downloaded the .dmg → double-clicked → installer opened → created admin credentials → clicked "Start and Show Splunk" → browser opened automatically.

Splunk Enterprise 10.4.0 running at localhost:8000. ✅

🔑 The solution that worked: 4 clicks. We spent hours on workarounds. The native solution was one tab-click away on the same download page. Always check the obvious options before going deep on workarounds.
PART 9

The First Real SOC Investigation

With Splunk running, clicked "Add Data" and uploaded secure.log — a Linux mail server security log from Splunk's official tutorial dataset.

Even the preview was alarming. Every single event was some variant of:

Failed password for invalid user [name] from [IP] port [number] ssh2

Someone had been hammering this mail server trying to get in via SSH, trying username after username. This is called a brute force attack.

Query 1 — Who was attacking and how many times?

source="secure.log" "Failed password"
| rex "from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count by src_ip
| sort -count

Result: 181 different IP addresses had been trying to break in. Top attacker — 87.194.216.51 — made 218 failed login attempts. Classic automated brute force: a script cycling through common usernames (guest, postgres, apache, mongodb, root, games, mail).

Query 2 — Did any of them get in?

This is the question that actually matters. Failed attempts are noise. Successful logins are incidents.

source="secure.log" "Accepted password"
| rex "for (?P<user>\w+) from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count by user src_ip
| sort -count

Result: 3 users with successful logins — djohnson, nsharpe, myuan. All from internal IPs: 10.3.10.46, 10.2.10.163, 10.1.10.172. Addresses starting with 10.x are private network addresses — inside the organization's network, not from the internet. These are legitimate employees.

The external attackers — every single one of them — failed.

Incident Report — Day 1

Detected SSH brute force attack against mailsv1 from 181 unique external IP addresses. Top attacker 87.194.216.51 made 218 failed authentication attempts. No external IP address achieved successful authentication. Internal users djohnson, nsharpe, and myuan show normal login activity from internal network addresses (10.x.x.x range). No escalation required. Recommend blocking top attacker IPs at perimeter firewall as precautionary measure.
That is a real SOC investigation. Detect the attack → quantify the scope → determine if it succeeded → write the summary. I did that on Day 1.
