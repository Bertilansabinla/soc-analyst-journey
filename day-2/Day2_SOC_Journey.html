A SIEM is not a product. It's four things happening in sequence:

StageWhat HappensSplunk Component
1. CollectDevices generate logs when events occurLog sources
2. TransportForwarder ships logs to SIEMUniversal Forwarder
3. ProcessNormalize + correlate patternsIndexer
4. PresentAnalyst queries + receives alertsSearch Head (web UI)
Key insight: On Day 1 I only worked at Stage 4. Today I understood the three stages that got data there. That changes how you think about every query you write.

Part 4 — The First Mistake: Time Range

I ran my first search and got zero events. Not because the data was missing — because the time picker defaulted to "Last 24 hours" and this incident happened months ago.

I changed it to "All time (real-time)" — which sounds right. It is not.

The trap: "All time (real-time)" searches for events arriving right now — not historical data. Completely different modes with confusing similar names.

The room instruction told me exactly what to do:

Once I selected the correct "All time" under Presets:

SOC Analyst lesson: Always check the time range before assuming data is missing. This is the #1 cause of unexpected zero results in Splunk. In a real incident this costs you time during an active attack.

Part 5 — The Ransomware Investigation

Two data sources: web_traffic logs and firewall_logs. My job: identify the attacker, trace the full attack chain, confirm data exfiltration. Five questions to answer.

Step 1 — Find the Attacker IP

I ran index=main sourcetype=web_traffic and clicked the client_ip field in the left panel to see top values.

One IP — 198.51.100.55 — made up 45.8% of ALL traffic while every other IP had just 3 events. That gap is not normal variance. That is an attack.

✅ Answer 1: Attacker IP = 198.51.100.55

Step 2 — When Did the Attack Peak?

index=main sourcetype=web_traffic
| timechart span=1d count | sort by count | reverse

Normal baseline ~500-600 events/day. Attack window: October 10-15. Peak on October 12 — 2,267 events, nearly 4x normal. The spike is the attacker being active. The return to baseline is them having achieved their goal.

✅ Answer 2: Peak day = 2025-10-12

Step 3 — What SQL Injection Tools Were Used?

sourcetype=web_traffic client_ip="198.51.100.55"
AND user_agent IN ("*sqlmap*", "*Havij*")
| stats count by user_agent
ToolEventsDescription
Havij/1.17993Automated SQL injection tool
sqlmap/1.7.9967Most popular SQL injection framework
✅ Answer 3: Havij count = 993

Step 4 — Path Traversal (including my wrong answer)

My first query caught the wrong attack type — redirect attacks, not path traversal:

I refined the query to isolate only the ../../ pattern:

sourcetype=web_traffic client_ip="198.51.100.55"
path="*../../*" | stats count by path
Lesson: Query precision matters. 633 vs 658 — close numbers measuring completely different attacks. Always verify what your filter is actually catching.
✅ Answer 4: Path traversal = 658

Step 5 — How Much Data Was Stolen? (Firewall Log Pivot)

This was the most important step. I switched from web traffic logs to the firewall logs — a completely different data source:

sourcetype=firewall_logs src_ip="10.10.1.5"
AND dest_ip="198.51.100.55" AND action="ALLOWED"
| stats sum(bytes_transferred) by src_ip
Key insight: Web logs told me WHAT the attacker did. Firewall logs told me WHAT LEFT the building. Neither alone tells the complete story. Connecting dots across sources — that is the actual SOC analyst skill.
✅ Answer 5: Bytes exfiltrated = 126,167

The Full Attack Chain

Phase 1: RECONNAISSANCE
→ Tools: curl, wget — probed /.env, /.git, /phpinfo

Phase 2: ENUMERATION
→ Path traversal: /download?file=../../etc/passwd (658 attempts)

Phase 3: SQL INJECTION
→ Havij/1.17 (993) + sqlmap/1.7.9 (967) = 1,960 total attempts

Phase 4: WEBSHELL + RANSOMWARE
→ shell.php uploaded via /upload.php
→ bunnylock.bin executed — full Remote Code Execution confirmed

Phase 5: C2 + EXFILTRATION
→ 10.10.1.5 called back to 198.51.100.55
→ 126,167 bytes stolen — action=ALLOWED — firewall did not block it
